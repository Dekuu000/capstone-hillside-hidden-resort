## 0. TL;DR

Hillside Hidden Resort is a **mobile-first, offline-capable PWA** for resort booking, payment verification, and QR-based check-in, with an admin operations console. It is a **Web3-hybrid** system: guest PII lives off-chain in Supabase, while privacy-safe proofs (escrow state, guest-pass NFTs, audit hashes) are mirrored to **Ethereum Sepolia** (Polygon Amoy retained as a future cutover target).

The stack is a **polyglot monorepo**:

- **`hillside-next/`** — Active Next.js 15 / React 19 PWA (the frontend). *This is the live app.*
- **`hillside-api/`** — FastAPI backend (trusted business logic, ~75 endpoints across 21 route groups).
- **`hillside-ai/`** — FastAPI ML microservice (Prophet/scikit-learn pricing & forecasting).
- **`hillside-contracts/`** — Solidity/Hardhat contracts (escrow + guest-pass NFT).
- **`supabase/`** — Postgres schema (109 migrations), RLS, RPCs, storage.
- **`packages/shared/`** — Shared Zod schemas/TypeScript types (the FE↔BE contract).
- ~~`hillside-app/`~~ — the earlier Vite/React prototype, **removed 2026-06-22** (superseded by `hillside-next/`; recoverable from git history).

## 3. Architecture

### 3.1 High-level shape

```
   Browser (Next.js PWA)
   │   • Supabase JS: email/password auth, Storage uploads, is_admin RPC
   │   • All business data via REST → API base URL (NEXT_PUBLIC_API_BASE_URL)
   │   • Offline: IndexedDB outbox/snapshots + hand-written service worker
   ▼
   FastAPI backend (hillside-api)   ── service-role Supabase client ──►  Supabase Postgres
   │   • Validates Supabase JWT (auth.get_user)                          (RLS, SECURITY DEFINER RPCs,
   │   • require_admin / require_authenticated / ownership guards         triggers, storage buckets)
   │   • Idempotency, payment policy, QR Ed25519 sign/verify
   ├──► web3 (lazy) ──► Ethereum Sepolia (EscrowLedger, GuestPassNFT)
   └──► HTTP (urllib) ──► hillside-ai (Prophet/sklearn) ── fallback to rule-based if unavailable
```

**Pattern:** Router → service/integration → external. The frontend is a **thin client**; Supabase is used by the browser almost exclusively for auth and file uploads, not for reading business data. The FastAPI layer is the trusted boundary where authorization, pricing, and chain interactions happen.

### 3.2 Frontend data flow

Two fetch wrappers split by execution context:

- **Client** — `lib/apiClient.ts` `apiFetch()`: prepends API base URL, sets `Authorization: Bearer <token>`, optional Zod validation, surfaces an "Offline" message when `navigator.onLine` is false.
- **Server (RSC)** — `lib/serverApi.ts` `fetchServerApiData()`: GET-only, uses Next `fetch` cache, `safeParse`, returns `null` on error so pages degrade gracefully instead of crashing. Reads the token from the `hs_at` cookie via `lib/serverAuth.ts`.

**Session model:** On login the Supabase access token is POSTed to `/api/auth/session` (a Next route handler) which sets httpOnly cookies `hs_at` (access token) and `hs_em` (email). SSR pages authorize via that cookie; the client SDK keeps the live Supabase session for token refresh and the offline sync loop.

### 3.3 Offline / Sync Center (the architecturally distinctive piece)

- `lib/offlineSync/store.ts` — IndexedDB (`hillside-offline-sync-v1`) with stores: entities, outbox, upload_queue, upload_blobs, sync_state, telemetry, conflicts, snapshots. Sensitive outbox payloads (`payment_submission`, `checkin`, `checkout`) are **AES-GCM encrypted at rest** (`lib/offlineSync/crypto.ts`; key in `localStorage`).
- `lib/offlineSync/engine.ts` — `runSyncCycle`: (1) upload queued blobs to Supabase Storage, (2) `POST /v2/sync/uploads/commit`, (3) `POST /v2/sync/push` (outbox replay with idempotency keys + conflict handling + backoff), (4) `GET /v2/sync/pull?cursor=` (delta pull). `startSyncLoop` ticks every ~15s and on the `online` event.
- `components/shared/SyncEngineProvider.tsx` — root context that boots the loop, picks scope (`admin` vs `me`) from the pathname, and drives the SyncCenter UI + banners.
- Snapshots cache bookings/dashboard/map/payments responses so pages render read-only while offline.
- QR: `lib/qrLocalVerify.ts` verifies guest QR Ed25519 signatures offline and tracks used JTIs to prevent replay during offline check-in.

### 3.4 Backend layering

- **Routers** (`app/api/v2/routes/*`) — HTTP concerns, auth dependencies, idempotency, orchestration (and meaningful business logic, e.g. rate computation in `reservations.py`).
- **Data access** — one large `app/integrations/supabase_client.py` (~3000 lines) wrapping all Supabase queries and DEFINER RPC calls.
- **Blockchain** — `escrow_chain.py` / `guest_pass_chain.py` (web3, lazy import); `core/chains.py` is a multi-chain registry resolving RPC/contract/signer from env.
- **AI** — `integrations/ai_pricing.py` POSTs to the external AI service; on any error returns deterministic rule-based fallbacks (AI is always non-blocking).
- **Config** — `core/config.py` pydantic-settings loaded from `hillside-api/.env`, single global `settings`.

## 5. Database Schema

Postgres 15. Enumerations are modeled as **TEXT + CHECK** (no `CREATE TYPE`). RLS is enabled on all ~24 tables; dominant pattern is *guest-owns-own + admin-sees-all*. ~30 `SECURITY DEFINER` RPCs implement atomic, anti-tamper operations.

### Core tables

| Table | PK | Key columns / notes |
| --- | --- | --- |
| `users` | `user_id` (→ `auth.users.id` CASCADE) | `role` CHECK **guest\|staff\|admin\|super_admin** (4-tier RBAC); `wallet_address` (regex-checked); role-escalation guard trigger |
| `units` | `unit_id` | `type` CHECK room\|cottage\|amenity; `operational_status` CHECK cleaned\|occupied\|maintenance\|dirty; `unit_code` UNIQUE; image arrays |
| `reservations` | `reservation_id` | `reservation_code` UNIQUE; `status` CHECK (incl. `no_show`); `balance_due` **GENERATED STORED**; shadow-escrow cols; guest-pass NFT cols; `reservation_source`, `sync_version`, deposit-policy cols; `cancellation_actor` (guest\|admin\|system) + `policy_outcome` (released\|refunded\|forfeited); promo cols (`original_total`, `discount_amount`, `promo_code`) |
| `reservation_units` | — | links reservations ↔ units |
| `services` / `service_bookings` | — | resort services and per-reservation bookings |
| `payments` | — | proof submissions + verification lifecycle |
| `checkin_logs` | — | check-in/out audit |
| `qr_tokens` | — | rotating signed QR tokens; **RLS fully locked (`USING false`)** — RPC/service-role only |
| `audit_logs` / `audit_anchors` | — | SHA-256 `data_hash` + `blockchain_tx_hash`/`anchor_id` (`audit_logs.anchor_id` → `audit_anchors`) |
| `ai_forecasts` / `ai_pricing_suggestions` / `ai_concierge_suggestions` | — | AI outputs |
| `resort_services` / `resort_service_requests` | — | concierge catalog + guest requests |
| `guest_welcome_notifications` | — | check-in welcome messages |
| `notifications` | `notification_id` | in-app notifications (`recipient_user_id`, `category`, `severity`, `read_at`, `dedupe_key`); RLS = recipient-only; retention-pruned when read + >90d |
| `promo_codes` / `promo_redemptions` | — | discount codes (percentage\|fixed, min-spend, caps, validity window, `auto_apply`) + per-use redemption tracking |
| `reviews` | — | guest reviews/ratings on completed bookings; `hidden_at` for admin moderation; drives live listing average |
| `sync_operation_receipts` / `sync_change_events` / `sync_upload_items` | — | offline sync (service-role only) |

**Relationships:** `reservations` is the hub — referenced by `reservation_units`, `service_bookings`, `payments`, `checkin_logs`, `qr_tokens`, `ai_pricing_suggestions`, `resort_service_requests`, `guest_welcome_notifications`. `users` is widely referenced (FK delete rules mix CASCADE/RESTRICT/SET NULL).

**Triggers:** auth-user provisioning (`handle_new_user`), `updated_at` bumpers, `bump_sync_version`, role-escalation guard (`prevent_non_admin_role_change`), reservation-status audit, sync changefeed emitters (`emit_sync_change_event`).

**RPCs (highlights):** `create_reservation_atomic` / `create_tour_reservation_atomic` (unit lock `FOR UPDATE NOWAIT` + server-side rate/total/deposit recompute + promo resolution — anti-double-book, anti-tamper), `resolve_promo_discount` (validate code or pick best auto-apply), `submit_payment_proof`, `perform_checkin`/`checkout`, `validate_qr_checkin`, `cancel_reservation` (cascades to service bookings), `get_available_units`, `get_reservation_quick_stats` (server-side count tiles). **Lifecycle RPCs (service-role only):** `release_expired_holds` (cancel unpaid holds past the 2h window, returns the true deadline), `mark_expired_no_shows` (forfeit deposit after grace), `prune_read_notifications` (delete read notifications older than the retention window).

**Storage buckets:** `payment-proofs` (private, per-user folder), `unit-images` (public read / admin write).

### Auth model details
- **JWT validation:** tokens are **not decoded locally** — `verify_access_token` calls Supabase `auth.get_user(token)` (cached 30s by token hash). `api_jwt_issuer`/`api_jwt_audience` config exists but is **unused**.
- **Role (4-tier RBAC):** resolved by querying the `users` table (cached 60s). `role_at_least` enforces the hierarchy guest < staff < admin < super_admin; `require_operations` = staff+, `require_admin` = admin+, and Team account management is super_admin-only. Ownership via `ensure_reservation_access`. A DB trigger blocks non-privileged role self-escalation.
- **Service-role key** is the default DB client (RLS-bypassing); a user-scoped variant attaches the token for select mutations.
- **QR signing:** Ed25519 primary (key from `QR_SIGNING_PRIVATE_KEY` or derived from legacy `QR_SIGNING_SECRET`), with legacy HMAC-SHA256 fallback. Tokens rotate (`qr_rotation_seconds`), are persisted server-side, and are **single-use**.

## 7. Key User Flows

### 7.1 Sign-up / Login
1. `app/register/page.tsx` or `app/login/page.tsx` → Supabase JS email/password auth (`lib/supabase.ts`).
2. Login upserts a `users` profile row directly via Supabase (`ensureUserProfileRow`, `login/page.tsx:142`).
3. Access token POSTed to `/api/auth/session` → sets httpOnly `hs_at`/`hs_em` cookies (`app/api/auth/session/route.ts`).
4. API context verified via `GET /v2/auth/context`; `resolveNextPath` routes admin → `/admin/reservations`, guest → `/my-bookings`.
5. Edge `middleware.ts` gates `/admin/*` and `/my-bookings` on cookie presence; `app/admin/layout.tsx` re-checks role server-side.

### 7.2 Guest booking → payment → confirmation (core flow)
1. **Browse/book:** `app/book` (`BookNowClient`) → SSR fetches `/v2/catalog/units/available`.
2. **Create reservation:** `POST /v2/reservations` → `routes/reservations.py` → `create_reservation_atomic` RPC (unit lock + server-side rate/deposit recompute). Status starts `pending_payment`. Optional escrow shadow-write + guest-pass NFT mint hooks fire.
3. **Pay deposit:** guest pays the manual method (e.g. GCash — `GcashPaymentGuide.tsx`), then submits proof via `POST /v2/payments/submissions` (idempotent, min-deposit checked). Offline: queued in IndexedDB outbox + blob, replayed by the sync engine.
4. **Admin verifies:** `app/admin/payments` → `POST /v2/payments/{id}/verify`. Reservation becomes eligible for QR/check-in.
5. **Stale holds:** a background scheduler (`release_expired_holds`, ~10-min tick) cancels unpaid `pending_payment` reservations ~2h after booking (outcome `released`), frees the held unit/slot, cascades to linked service bookings, and notifies the guest.
6. **No-show:** confirmed bookings past check-out with no check-in are flagged `no_show` (deposit `forfeited`) after a grace period by `mark_expired_no_shows`; staff can also mark a no-show manually. Promo discounts are applied to the total before the deposit (20%, floor ₱500, cap ₱5,000) is computed.

### 7.3 QR check-in (admin)
1. Guest opens QR pass: `app/guest/my-stay` → `POST /v2/qr/issue` (rotating Ed25519 token), rendered with `qrcode.react`.
2. Admin scans: `app/admin/check-in` (`AdminCheckinClient`, html5-qrcode) → `POST /v2/qr/verify` validates expiry, signature, single-use consume, reservation/code match. Offline: `lib/qrLocalVerify.ts` verifies the signature locally and tracks used JTIs.
3. `POST /v2/checkins` → `perform_checkin` RPC; syncs unit → `occupied`, releases escrow, generates welcome notification (non-blocking).
4. Check-out: `POST /v2/checkouts` requires `balance_due == 0` (409 otherwise); unit → `dirty`.
5. Demo override: `feature_checkin_schedule_bypass` can bypass the "check-in only on reservation date" gate.

### 7.4 Admin operations
- Reservations console (`/admin/reservations`), walk-in creation, unit inventory, payment review, service queue, reports/CSV export, blockchain/escrow reconciliation, AI pricing center. All admin routes require `role == admin` (enforced in `app/admin/layout.tsx` + `require_admin` on the API).

## 8. Smart Contracts (Sepolia / Amoy)

Solidity `0.8.24`, Hardhat + ethers v6. Single deployer EOA key (`DEPLOYER_PRIVATE_KEY`) is operator/owner/minter.

- **`EscrowLedger.sol`** — native-currency booking escrow. `mapping(bytes32 bookingId => EscrowRecord)`, `enum EscrowState {None, Locked, Released, Refunded}`. `lock(bookingId, recipient)` payable (anyone); `release` / `refund` are `onlyOperator`. `bookingId = keccak(reservation_id)`. Backend mirrors state with a "shadow-write" mode (fake `shadow-` tx hashes) when on-chain locking is flagged off.
- **`GuestPassNFT.sol`** — hand-rolled minimal ERC-721. Stores only `reservationHash` on-chain (no PII). `mintGuestPass(recipient, reservationHash)` is `onlyMinter`, rejects duplicate reservation hashes.

A background **escrow reconciliation monitor** (`observability/escrow_reconciliation_monitor.py`) periodically compares DB vs on-chain escrow state and raises alerts on mismatch.

## 9. AI Service (`hillside-ai`)

Single-file FastAPI app. Endpoints: `GET /health`, `POST /v1/pricing/recommendation` (sklearn **Ridge** on a synthetic training set), `POST /v1/occupancy/forecast` (**Prophet**, cached 300s), `POST /v1/concierge/recommendation` (sklearn **NearestNeighbors**). Every path has a graceful fallback chain down to deterministic rule-based output if the ML lib is unavailable. Models are trained on synthetic/in-request data at runtime — **demo-grade, not production ML** (comments explicitly note "for capstone defense panel").

## 10. Issues, Gaps & Risks

### Security / authorization
- **Forgeable audit logs (DB):** `audit_logs` INSERT policy is `WITH CHECK (true)` (`supabase/migrations/20260207006_audit_logs.sql`) — any authenticated user can insert arbitrary audit rows, which are then hashed and intended for blockchain anchoring. **High concern** given the integrity claims.
- **Broad reservation UPDATE policy:** `owners_or_admins_update_reservations` lets a guest update arbitrary columns on their own reservation at the RLS layer; integrity relies on clients using DEFINER RPCs instead of direct writes.
- **Service-role key is the default DB client** (`integrations/supabase_client.py`): most reads/writes bypass RLS, so security depends entirely on app-layer `require_admin` / `ensure_reservation_access` being correct on every route. Large blast radius (no missing check was found, but the surface is wide).
- **`search_path` not pinned on later DEFINER functions** (`20260418002/003/006`) — a latent privilege-escalation surface; only partially remediated.
- **Error detail leakage:** the 500 handler and several routes return raw `str(exc)` (`main.py:139-152`; `qr.py:146,183,209`; `ai.py:322`) — can leak DB/web3 internals in production.
- **CORS + credentials:** `allow_credentials=True` with `allow_methods/headers=["*"]` is safe only because origins are an explicit allowlist; no guard rejects a wildcard origin if misconfigured.
- **Webhook auth opt-in:** if `payment_mode="gateway"` is enabled with no token/secret set, `/v2/payments/webhooks/provider` is unauthenticated (default `proof_only` mitigates).
- **AI service is unauthenticated** (no auth/CORS/rate-limit on any endpoint) — fine on localhost, a DoS/compute surface if exposed.
- **Unbounded in-process auth/role caches** (`auth.py`) grow without eviction; also a role demotion / token revocation can lag up to 60s.

### Correctness / fragility
- **Status enum drift:** `BOOKING_STATUSES` in `packages/shared/src/types.ts` includes `draft`/`escrow_locked`, which the DB `reservations.status` CHECK does **not** allow — a TS-valid value would be rejected by Postgres.
- **`EscrowLedger.release/refund` use `.transfer()`** (2300-gas stipend) — reverts for contract-wallet recipients; prefer `.call`.
- **Unbounded `lock()`** — anyone can occupy a `bookingId` slot (griefing/front-run); no payer↔booking authorization.
- **Near-zero contract test coverage** — only one happy-path `EscrowLedger` emit test; no tests for `release`/`refund`/access control or any of `GuestPassNFT`.
- **Synchronous on-chain waits inside request handlers** (`escrow_chain.py:287`) — `wait_for_transaction_receipt` (up to 90s) can tie up worker threads when on-chain locking is enabled.
- **O(n) table scans / Python-side filtering** in several admin/guest list paths (e.g. `list_admin_payments` scans up to 5000 rows; reservation search `range(0,999)`) — won't scale.
- **RPC churn:** `create_reservation_atomic`/`create_tour_reservation_atomic` recreated across ~9 migrations each; signature changes risk lingering ambiguous overloads (a dedicated `drop_legacy_overloads` migration exists for exactly this).
- **3000-line `supabase_client.py`** — single-file data layer is a maintainability risk.

### Incomplete / placeholder
- **Google sign-in is a stub** (`login/page.tsx:281`) yet a prominent "Continue with Google" button is shown — misleading UX.
- **Missing routes:** landing footer links `/about-us` and `/contact-us` (`app/page.tsx:431`) 404.
- **Escrow release-tx not wired:** only TODO in the frontend (`components/admin-escrow/AdminEscrowTableClient.tsx:137`) — backend exposes one `chain_tx_hash`, no `release_tx_hash`.
- **Stale health label:** `app/api/health/route.ts:6` still returns `mode: "foundation-shell"`.
- **Audit on-chain anchoring not produced:** `audit_logs.anchor_id`/`blockchain_tx_hash` are read/filtered but never written by the API — anchoring is modeled but not implemented.
- **Landing imagery/prices are hardcoded** Unsplash URLs + static placeholders, not live data.

### Cross-cutting
- The **escrow state machine is duplicated across four layers** (Solidity enum, DB CHECK, shared TS type, reconciliation logic) with no single source of truth enforcing agreement.
- **`hillside-app/` legacy app** was removed on 2026-06-22 (superseded by `hillside-next/`); recoverable from git history if a before/after comparison is needed.
- No `TODO/FIXME` markers exist in the backend `app/`, contracts, AI service, or migrations; the frontend has exactly one (escrow release-tx).
